# Agent Observability Design (Local Logs + LangFuse)

**Date:** 2026-03-03  
**Status:** Approved  
**Scope:** LLM-agent implementation only (`src/overcooked_ai_py/agents/llm/*` + `scripts/run_llm_agent.py`)

## Overview

Add observability in two complementary paths:

1. Local per-run structured logs (one file per run)
2. LangFuse tracing for LLM/tool visibility, usage, and cost tracking

This design intentionally avoids touching the core Overcooked framework and focuses on the custom LLM-agent layer.

## Requirements

1. Cover both agent modes:
   - `llm`
   - `planner-worker`
2. Enforce mode-specific tagging in both local logs and LangFuse.
3. Create a new local log file per run (no append-only global file).
4. Support benchmarking metadata (run name/title/tags/experiment/variant/notes).
5. LangFuse failures (missing keys, network/SDK issues) must not break runs.
6. Keep integration broad and reusable for mass experimentation.

## Approaches Considered

### Approach A: Graph-level instrumentation (Recommended)

- Primary hook: shared graph invocation path (`graph_builder.py`) plus small parity hook for legacy `graph.py`
- Benefits:
  - Captures each LLM call and tool call with minimal duplication
  - Keeps planner/worker instrumentation consistent
  - Lowest long-term maintenance burden
- Trade-off:
  - Requires light plumbing of observability context into graph invoke calls

### Approach B: Agent-level instrumentation only

- Hooks only in `LLMAgent`, `Planner`, `WorkerAgent`
- Benefits:
  - Simple conceptually
- Trade-off:
  - Misses detailed LLM/tool loop events unless duplicated across files

### Approach C: Runner-only instrumentation

- Hooks only in `scripts/run_llm_agent.py`
- Benefits:
  - Minimal edits
- Trade-off:
  - Poor internal visibility for reasoning/tool execution

## Final Architecture

Add `src/overcooked_ai_py/agents/llm/observability.py` with:

1. `RunContext`
   - `run_id`, `run_name`, `run_title`, `mode`, `layout`, `model`, `experiment`, `variant`, `tags`, `notes`
2. `ObservabilitySink`
   - Single `emit(...)` API
   - Fan-out to local file logger and optional LangFuse reporter
3. `FileRunLogger`
   - Writes JSONL in `logs/agent_runs/`
4. `LangFuseReporter`
   - Optional, graceful no-op on init/reporting failure

### Trace and Event Taxonomy (Locked)

LangFuse and local JSONL will mirror the same structure:

- `run.start` / `run.end`
- `llm.generation`
- `tool.call`
- `planner.assignment`
- `action.commit`
- `error`

Interpretation:

1. One **LangFuse trace per episode run**.
2. Each LLM invocation is a **generation** observation.
3. Each tool execution is a **tool** observation.
4. Planner task assignment is a dedicated planner assignment event/span.
5. Final selected action per agent-step is an action commit event/span.

## Local Log Design

### Location and Naming

- Directory: `logs/agent_runs/`
- One file per run, JSONL format
- Filename:
  - `{timestamp}_{mode}_{layout}_{run_name_slug}_{run_id}.jsonl`

### Common Event Fields

Each JSONL line includes:

- `ts`
- `run_id`
- `run_name`
- `run_title`
- `mode`
- `layout`
- `model`
- `experiment`
- `variant`
- `tags`
- `step` (nullable)
- `agent_role` (e.g., `planner`, `worker_0`, `worker_1`, `llm_agent`, `runner`)
- `event_type`
- `payload` (event-specific data)

## LangFuse Integration Details

Reference docs:

- [LangChain/LangGraph integration](https://langfuse.com/integrations/frameworks/langchain)
- [LangGraph cookbook](https://langfuse.com/guides/cookbook/integration_langgraph)
- [LangFuse Python SDK reference](https://python.reference.langfuse.com/langfuse)

Planned behavior:

1. Initialize LangFuse once in runner when credentials are available.
2. Use `CallbackHandler` and pass via `config={"callbacks": [...]}` on invoke.
3. Apply dynamic trace attributes:
   - `langfuse_session_id` (run id)
   - `langfuse_tags` (includes `mode:llm` or `mode:planner-worker`)
   - metadata fields (experiment, variant, layout, run title, notes)
4. On script completion, call shutdown/flush for short-lived process reliability.
5. If LangFuse cannot initialize, continue with local logs only.

## CLI Metadata for Benchmarking

Add observability-specific flags in `scripts/run_llm_agent.py`:

- `--run-name` (default autogenerated)
- `--run-title` (default empty)
- `--tags` (comma-separated; default auto-tags include mode + layout)
- `--experiment` (default `default-exp`)
- `--variant` (default `baseline`)
- `--notes` (default empty)

Mode tags are always enforced:

- `mode:llm` for legacy mode
- `mode:planner-worker` for planner-worker mode

## Reliability and Failure Policy

1. Observability must be best-effort; never break gameplay loop.
2. Logging exceptions are swallowed after optional debug print.
3. LangFuse is optional and auto-disabled on setup/runtime errors.
4. Local JSONL writer opens in append mode per run file and flushes frequently.

## Validation Plan

1. Unit tests:
   - per-run file creation and valid JSONL records
   - mode tag enforcement
   - no-crash behavior when LangFuse unavailable
   - invoke callback/config propagation
2. Manual smoke:
   - run `--agent-type llm` and `--agent-type planner-worker`
   - verify distinct mode tags in logs/traces
   - verify run continues with LangFuse disabled/misconfigured
